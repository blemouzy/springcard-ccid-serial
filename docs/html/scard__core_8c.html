LONG SCARD_Connect ( BYTE  bSlot,
BYTE  abAtr[],
DWORD *  pdwAtrLength 
)

Connect to the card in the given slot (if some)

Note
This is not exactly the same prototype as SCardConnect in the PC/SC standard, but it provides the same feature
Parameters
bSlotslot number (0 for the contactless slot, 1 for the 1st contact slot, etc)
abAtrbuffer to receive the ATR (must be at least 32-byte long)
pdwAtrLengthIN: the size of the ATR buffer; OUT: the actual length of the ATR
Returns
SCARD_S_SUCCESS success, ATR is valid
SCARD_W_REMOVED_CARD: there's no card in the slot
SCARD_W_UNSUPPORTED_CARD: there's a card in the slot, but its ATR is invalid
SCARD_W_UNRESPONSIVE_CARD: there's a card in the slot, but it is mute
Other code if internal or communication error has occured
Note
This function is based on CCID PC_TO_RDR_IccPowerOn

◆ SCARD_Status()

LONG SCARD_Status ( BYTE  bSlot,
BOOL *  pfCardPresent,
BOOL *  pfCardPowered 
)

Retrieve the status of a slot:

  • is there a card in the slot or not?
  • is the card powered or not?
    Note
    This is not exactly the same prototype as SCardStatus in the PC/SC standard, but it provides the same information
    Parameters
    bSlotslot number (0 for the contactless slot, 1 for the 1st contact slot, etc)
    pfCardPresentpointer to receive the 'present' flag
    pfCardPoweredpointer to receive the 'powered' flag
    Returns
    SCARD_ERR_S_SUCCESS success, flags are reliable
    Other code if internal or communication error has occured
    Note
    This function is based on CCID PC_TO_RDR_GetSlotStatus
    There's no need to loop around SCARD_Status to wait for card insertion. Looping around SCARD_Connect is more efficient since when a card is inserted, the ATR will be returned at once. On the other hand, looping around SCARD_Status is useful to wait for card removal after a call to SCARD_Disconnect.
    See also
    SCARD_Connect
